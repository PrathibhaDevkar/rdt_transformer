Core Idea

What is a Recurrent Depth Transformer?

Standard transformers stack N unique layers. The RDT replaces most of them with one shared block applied N times — iterative refinement with a fraction of the parameters.

The Core Idea

A standard GPT needs N unique transformer blocks for N layers of reasoning. The RDT achieves the same effective depth using one block applied N times — sharing weights across all loop steps.

At each loop step the model refines its hidden state h using the update rule on the right. The original encoded input e is re-injected every step as an anchor so the model never loses track of what was originally asked.

At inference you can run more loops on harder questions, getting more reasoning compute with zero retraining and no new parameters.

Core Update Rule — Applied Each Loop Step
h(t+1) = A·h(t) + B·e + Transformer(h(t))
h(t)Hidden state at loop step t — the model's current "draft" answer, starts as encoded input
ALearned N_EMBED × N_EMBED matrix — controls how much of the previous state carries forward
BLearned N_EMBED × N_EMBED matrix — controls how much the original input anchors each step
eEncoded input from Prelude (fixed) — re-injected every loop step to prevent drift
Stability constraint: The spectral radius of A is capped at 0.9 after every gradient step via SVD projection — preventing h from diverging across loop iterations.

Key Innovations

What Makes This Implementation Unique

Six design choices that make the RDT work — each addressing a specific failure mode in naive recurrent transformers.

🔄
Shared Recurrent Weights

One transformer block is applied N times with identical weights. The hidden state h(t) is iteratively refined, giving the model more compute per parameter than a standard stacked architecture.

📐
Spectral Radius Enforcement

After every gradient step, SVD is computed on A and its spectral radius is capped at 0.9. This keeps the hidden state from diverging across loop iterations and makes training reliably stable.

⚓
Input Re-injection via B·e

The original encoded input e is re-injected at every loop step via the B matrix. This keeps the model grounded in the original question and prevents drift as reasoning deepens.

🔗
LM Head Weight Tying

The LM head weight matrix is shared with the token embedding table. Since the embedding maps IDs to vectors and the head maps vectors back to IDs, sharing saves ~19M parameters with no accuracy loss.

📚
Curriculum Learning

Models are trained stage-by-stage from 1-hop to 5-hop chains, only advancing when accuracy exceeds 82%. Starting simple and building up prevents the model from learning shortcuts on hard examples.

⚡
Inference-Time Compute Scaling

At inference the number of loop steps can be increased for harder problems, with no retraining and no new weights. A model trained with 6 loops can run 14 loops on a hard question at test time.

Future Scope

Where This Goes Next

The RDT architecture opens multiple interesting directions for research and production application.

🧠
Adaptive Loop Depth

Train a small effort predictor that decides how many loop steps to allocate per input. Easy questions get fewer steps, hard ones get more, with dynamic compute allocation that needs no user tuning.

🔢
Scale to GPT-2 Size

Apply the RDT at 124M-param scale. The parameter savings compound at larger sizes, so a 124M-param RDT could realistically match a 200M+ GPT on reasoning benchmarks while using significantly less memory.

🌐
Multi-Modal RDT

Extend the recurrent block to handle interleaved text and image tokens. The iterative refinement mechanism is modality-agnostic and could improve visual reasoning and image captioning tasks.

🔍
Loop Step Interpretability

Probe what each loop step computes by analyzing hidden state evolution across iterations. Early steps likely resolve syntax while later steps handle semantic reasoning, similar to GPT layer probing work.

🤖
RLHF Fine-tuning

Apply reinforcement learning from human or AI feedback on top of the pretrained RDT. The chain-of-thought traces generated in Phase 3 are a solid starting point for reward modeling and preference learning.

📦
Drop-in Hugging Face Block

Wrap the RecurrentBlock as a replacement for standard transformer layers. Any existing GPT-style training pipeline could pick up recurrent depth with just a few lines of code changed.

Get Started

Run It Yourself

Python 3.9+ and PyTorch 2.0+. MPS (Apple Silicon), CUDA, and CPU are all auto-detected.

1 — Install
# clone the repo git clone https://github.com/PrathibhaDevkar/rdt_transformer.git cd rdt_transformer # install dependencies pip install -r requirements.txt
2 — Phase 1: GPT Baseline
python phase1/lesson1_tokenizer.py python phase1/lesson2_embeddings.py python phase1/lesson3_attention.py python phase1/lesson4_feedforward.py python phase1/lesson5_training.py # trains → best_model.pt
3 — Phase 2: RDT + Benchmark
python phase2/lesson1_rdt.py # trains RDT python phase2/lesson2_benchmark.py # GPT vs RDT python phase2/lesson2c_curriculum.py # curriculum
4 — Phase 3: Fine-tune + Agent
python phase3/lesson1_finetune.py # tool-use QA python phase3/lesson1c_cot.py # chain-of-thought python phase3/lesson2_agent.py # agent loop
Note: The dataset (data/shakespeare.txt) is downloaded automatically on first run. Model checkpoints are saved to results/.
